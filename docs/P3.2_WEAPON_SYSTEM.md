# P3.2 — Data-driven weapon system + switching

> Second milestone of Phase 3. Goal: turn the single hard-coded rifle into a **`WEAPONS` data table** with **Rifle / Shotgun / SMG**, switchable by number key, each with its **own ammo**. Weapon identity reads from **HUD + projectile behaviour, never character art** — the soldier keeps holding the same rifle. The enabling refactor: **bullets carry their own damage/range/tint, stamped at `fire()` time**, so a weapon switch never reaches back and mutates a round already in flight.

## Context
Today there is exactly one weapon. `CONFIG.weapon` is a flat object (`name, bulletSpeed, fireRate, magSize, reloadTime, bulletRange, damage, bulletPoolSize`) and every consumer reads that global directly:
- `Player.js` — inits `this.ammo = CONFIG.weapon.magSize`, gates fire on `weapon.fireRate`, reloads to `weapon.magSize`, reads `CONFIG.weapon.bulletSpeed` in `spawnBullet`, decrements a single `this.ammo`.
- `Bullet.js` — expires on the **global** `CONFIG.weapon.bulletRange` in `preUpdate`.
- `GameScene.js` — sizes the pool from `CONFIG.weapon.bulletPoolSize`; deals the **global** `CONFIG.weapon.damage` at collision in `onBulletHitEnemy`.
- `UIScene.js` — shows `CONFIG.weapon.name` and `CONFIG.weapon.magSize`.

Two of those (damage, range) are resolved **globally at the wrong moment** — damage at the instant of collision, range mid-flight. The moment two different projectiles are alive at once (a shotgun pellet and a rifle round; or a bullet still travelling after you've switched guns), the global lookup is wrong. So the real core of P3.2 is not "add a table" — it's **move per-shot properties onto the projectile**. Once done, a fourth weapon is a data row, and P3.3's upgrades (which mutate weapon stats) flow through for free because damage is resolved from the live weapon at fire time and then carried by the bullet.

## Scope guard
**In P3.2:**
- A `WEAPONS` table in config with **three** rows — **Rifle** (`fireMode:'auto'`), **Shotgun** (`fireMode:'single'`, multi-pellet spread), **SMG** (`fireMode:'auto'`, fast + low-damage). Rifle's row reproduces today's values **exactly** (regression-safe).
- **Per-weapon current-mag ammo** on the player; reload refills only the **active** weapon's mag.
- **Switching by number key** `1 / 2 / 3`. Switch cancels any in-progress reload; the fire cooldown is shared across weapons (no switch-to-bypass-fire-rate exploit).
- **Projectile refactor:** `bullet.fire(x, y, vx, vy, { damage, range, tint })` stamps damage/range/tint onto the bullet; `preUpdate` expires on `this.range`; `onBulletHitEnemy` deals `bullet.damage`. **A bullet in flight is unaffected by a mid-air weapon switch.**
- **Shotgun spread:** one trigger pull = **one ammo** = **N pellets** fanned within `spreadDeg` around the aim vector.
- **Weapon identity = HUD + projectile:** HUD shows the current weapon (text label + ammo now; icon slot reserved) and per-weapon projectile look (`projectileTint`) and muzzle scale (`muzzleScale`). **No character-art change** — the Soldier_1 rifle pose is used for all three.
- Shared **projectile pool** resized for the worst case (shotgun burst + SMG spray).

**NOT in P3.2:**
- Scrap / shop / upgrades (P3.3) — but leave the damage-on-bullet seam that P3.3 plugs into.
- Reserve ammo / `ammoByType` pool and save (P3.5) — P3.2 tracks **current mag per weapon only**; reserve is effectively infinite for now. `ammoType` may sit in the data as a reserved field, unenforced.
- Per-weapon **fire/reload SFX** (P3.9 audio) — `sfxFire`/`sfxReload` may sit in the data as reserved fields, inert until P3.9.
- **`fireMode:'burst'`** — a valid schema value with **no consumer** in this set (rifle/SMG=auto, shotgun=single). Do not build the burst path.
- Real **HUD weapon icons** (a text label ships now; a ~32px icon is a later art drop via the swap-point).
- Scroll-wheel switching (number keys are the spec; wheel is a deferrable nicety).
- New enemy types, terrain, save, menus.

## Design decisions
- **`WEAPONS` is the single source of truth, read live.** The player holds a `currentWeaponId` (string), **not** a copy of the stats. Fire-time code does `const w = CONFIG.WEAPONS[this.currentWeaponId]` and reads from that. **Never snapshot a weapon's stats onto the player** — a snapshot would go stale the moment P3.3 upgrades mutate a weapon row. This is the pattern P3.3 depends on.
- **Bullets are self-contained (the headline refactor).** `fire()` gains a stats object; the bullet stamps `this.damage`, `this.range`, and applies `this.setTint(tint)` (or clears it). `preUpdate` compares against **`this.range`**; the overlap callback deals **`bullet.damage`**. Result: switching weapons, or firing two weapons' rounds back-to-back, never cross-contaminates a projectile mid-flight. This is asserted directly in verification (test **G**).
- **One trigger pull = one ammo = `pellets` projectiles.** Ammo decrements **once per shot** (moves out of the old per-bullet spot), then a loop spawns `pellets` bullets. Rifle/SMG use `pellets:1, spreadDeg:0` (a single straight round — identical to today for the rifle). Shotgun uses `pellets:N` with each pellet's velocity rotated by a random offset within `±spreadDeg/2` of the aim vector. Muzzle flash + camera shake + the shoot anim fire **once per shot**, not per pellet.
- **Fire modes: `auto` and `single` only.**
  - `auto` (rifle, SMG) = today's behaviour: hold the button, fire on the `1/fireRate` cadence.
  - `single` (shotgun) = fire on the pointer **press edge** (Phaser `pointer` down-this-frame), not while held — so a pump shotgun needs a click per shell. This is the one genuinely new input branch.
  - `burst` — deferred; not wired.
- **Per-weapon ammo = current mag, keyed by weapon id.** `this.ammo` becomes `{ rifle: n, shotgun: n, smg: n }` (or a `Map`), each initialised to its weapon's `magSize`. Fire/reload/HUD all read the active id. Reserve ammo is out of scope (P3.5); mag is the only gate now.
- **Switching (`1/2/3`):**
  - Maps ordered ids `['rifle','shotgun','smg']` to keys ONE/TWO/THREE; `JustDown` → `switchWeapon(id)`.
  - Switching to the already-active weapon is a no-op.
  - Switching **cancels an in-progress reload** (no refund) and interrupts nothing else.
  - The **fire cooldown (`shootTimer`) is shared** and carries across the switch, so you can't spam-switch to dodge fire-rate.
  - Disabled while `dead`/`won`.
- **Weapon identity through HUD + projectile, per PHASE3_PLAN Decision 1/2.** The soldier always visually holds the rifle. What changes per weapon: the **HUD** (name label + that weapon's mag/ammo; an icon slot reserved for later), the **projectile** (`projectileTint`, `bulletSpeed`, count/spread), and the **muzzle flash scale** (`muzzleScale`). No per-weapon character sheet.
- **Shared projectile pool.** Conceptually it's no longer "the rifle's pool." Move the size to a top-level `CONFIG.bulletPoolSize` and raise it to cover the worst case — an SMG stream (high `fireRate`, in-flight lifetime ≈ `range/bulletSpeed`) plus a shotgun volley. Target ~48; confirm under the FPS/leak test.
- **Placeholder-first (golden rule 4).** Projectiles differ by **tint** on the existing bullet texture (no new art needed); HUD weapon shows a **text label** now. Real per-weapon HUD icons (and later per-weapon SFX) drop in through the swap-point with zero logic change.
  - **On tint (re: the P3.1 follow-up note):** P3.1 logged that the enemy hurt-flash `setTint(0xffffff)` is a no-op — that is **white-specific** (white is the multiply identity), not a general tint failure. Our projectile tints are **non-white** (`0xffd27f`, `0x9ad0ff`) over the **light** `placeholder-bullet` texture (`#e8e8c0`), so multiply-tint colorizes correctly. If a **dark** bullet sprite is ever swapped in, colorize with `setTintFill` (or keep a light base) — but for now `setTint` is right. (The hurt-flash fix itself stays a parked P3.1 follow-up, out of P3.2 scope.)

## Changes by file
1. **`src/config.js`**
   - Replace the flat `weapon: { … }` block with a **`WEAPONS` table** and a `defaultWeaponId`:
     ```
     WEAPONS: {
       rifle:   { name:'RIFLE',   fireMode:'auto',   damage:12, fireRate:6,  magSize:12, reloadTime:1.1, bulletSpeed:1240, range:800, pellets:1, spreadDeg:0,  ammoType:'rifle',   projectileTint:null,     muzzleScale:1.0, hudIcon:null, sfxFire:null, sfxReload:null },
       shotgun: { name:'SHOTGUN', fireMode:'single', damage:7,  fireRate:1.5, magSize:6,  reloadTime:1.6, bulletSpeed:1000, range:420, pellets:7, spreadDeg:18, ammoType:'shell',  projectileTint:0xffd27f, muzzleScale:1.4, hudIcon:null, sfxFire:null, sfxReload:null },
       smg:     { name:'SMG',     fireMode:'auto',   damage:7,  fireRate:14, magSize:30, reloadTime:1.3, bulletSpeed:1180, range:640, pellets:1, spreadDeg:3,  ammoType:'smg',     projectileTint:0x9ad0ff, muzzleScale:0.8, hudIcon:null, sfxFire:null, sfxReload:null },
     },
     defaultWeaponId: 'rifle',
     ```
     *(All numbers are tuning starting points — rifle intentionally == today's values; balance in playtest. `ammoType`/`sfx*`/`hudIcon` are reserved fields, unenforced this milestone.)*
   - Add top-level `bulletPoolSize: 48` (shared pool; was `weapon.bulletPoolSize: 30`).
   - `muzzleOffset` stays global (same gun art for all weapons).
   - Leave `TEXTURE_MAP.bullet` as-is; projectile difference is tint at fire time, not a new key.
2. **`src/entities/Bullet.js`**
   - Change `fire(x, y, vx, vy)` → `fire(x, y, vx, vy, stats)` where `stats = { damage, range, tint }`. Stamp `this.damage = stats.damage; this.range = stats.range;` and `stats.tint == null ? this.clearTint() : this.setTint(stats.tint)`.
   - In `preUpdate`, expire on **`this.distanceTraveled >= this.range`** (not `CONFIG.weapon.bulletRange`).
   - `deactivate()` also `this.clearTint()` so a pooled bullet doesn't inherit a stale tint.
3. **`src/entities/Player.js`**
   - Add weapon state: `this.currentWeaponId = CONFIG.defaultWeaponId;` and per-weapon mags `this.ammo = { rifle: …magSize, shotgun: …, smg: … }` built from `CONFIG.WEAPONS`. Add ONE/TWO/THREE keys; keep the R reload key.
   - Add a `get weapon()` helper → `CONFIG.WEAPONS[this.currentWeaponId]` (live lookup). Add `switchWeapon(id)` (no-op if same/dead/won; cancels reload; leaves shared `shootTimer`).
   - In `update`: read weapon-switch keys (`JustDown` on 1/2/3). Replace the fire gate with a **fire-mode branch** — `auto`: `pointer.isDown`; `single`: `Input.Pointer` **just-down** this frame. Both still gated on `shootTimer<=0 && mag>0 && !reloading && aimingForward`.
   - Rename `spawnBullet()` → `fireWeapon()`: decrement the **active weapon's mag once**, then loop `weapon.pellets` — for each, rotate the base aim velocity by a random `±spreadDeg/2`, compute `vx/vy` at `weapon.bulletSpeed`, `bullet.fire(mx, my, vx, vy, { damage: weapon.damage, range: weapon.range, tint: weapon.projectileTint })`. Muzzle particle (scaled by `weapon.muzzleScale`), `player-shoot` anim, and camera shake fire **once**, outside the pellet loop.
   - Reload refills **`this.ammo[currentWeaponId]`** to `weapon.magSize` over `weapon.reloadTime`; gate on active mag `< magSize`.
4. **`src/scenes/GameScene.js`**
   - Pool: `maxSize: CONFIG.bulletPoolSize`.
   - `onBulletHitEnemy`: `enemy.takeDamage(bullet.damage)` (was `CONFIG.weapon.damage`).
5. **`src/scenes/UIScene.js`**
   - Read the **active** weapon from the player: `const w = gameScene.player.weapon;` show `w.name` (weapon label) and `AMMO {player.ammo[player.currentWeaponId]} / {w.magSize}` (or the reloading readout). Reserve a spot for a future weapon icon (text label for now). Debug bullet-count line unchanged.

## Verification (harness + human)
Reuse the puppeteer-core + Chrome driver at :8081.
- **A. Build/console:** `npm run build-nolog` clean; 0 errors/404s.
- **B. Rifle regression (the safety net):** with the rifle equipped, one trigger = one bullet, `damage 12`, `range 800`, `fireRate 6`, `magSize 12`, `reload 1.1s` — assert a rifle bullet's stamped `damage`/`range` match the old constants and behaviour is byte-for-byte the old feel.
- **C. Data-driven:** all three weapons resolve from `CONFIG.WEAPONS`; adding a throwaway 4th **data row** makes it switchable/fireable with **no engine edit** (spot-check, then remove).
- **D. Shotgun spread:** one `single`-mode trigger pull consumes **1** ammo and spawns exactly `pellets` active bullets within `±spreadDeg/2`; holding the button does **not** auto-fire; a pellet's stamped `damage` == shotgun damage (≠ rifle).
- **E. SMG:** `auto`, higher `fireRate` + lower `damage` than rifle; holding sprays; mag drains at the SMG rate.
- **F. Switching + per-weapon ammo:** `1/2/3` change `currentWeaponId` and the HUD label/ammo; each weapon's mag is preserved independently across switches; switching **cancels an in-progress reload**; rapid switch-fire cannot beat the shared `shootTimer` cadence.
- **G. In-flight independence (the refactor's whole point):** fire a rifle round, switch to shotgun **before it lands**, assert the in-flight bullet still deals **rifle** damage and expires at **rifle** range — a mid-air switch does not mutate it.
- **H. Pool + FPS:** SMG spraying + shotgun volleys; projectiles are reused (no unbounded allocation), no leaked active bullets after they expire, `actualFps ≥ 55`.
- **I. Deliverable:** screenshots (each weapon's tinted projectile, the shotgun fan, the HUD per weapon) + a pass/fail table. Then human playtest: the three weapons must **feel** distinct (rifle steady, shotgun punchy/close-range, SMG spray) and switching must be instant/legible. On pass: add a **P3.2 row to `PROGRESS.md`** under "Phase 3 — game systems" (matching the P3.1 entry — what it delivered + any parked follow-ups), then `git commit -m "P3.2: data-driven weapon system (rifle/shotgun/smg) + switching + per-bullet damage/range"`.
