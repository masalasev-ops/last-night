# P3.4 — Enemy roster expansion (runner / tank / flyer)

> Fourth milestone of Phase 3. Add **three** enemy types so a fight has texture: a fast, fragile **Runner**, a slow, heavy **Tank**, and (the one that needs code) a gravity-ignoring **Flyer** that homes in 2D and can reach a perched player. All three are in scope — none deferred. The enabling refactor is small and specific: today only `enemyProfile:'ranged'` reads per-type stats from its `ENEMIES` row — **melee types all read the shared `CONFIG.enemy` block**. Generalize that so *any* type resolves `this.def.<stat> ?? CONFIG.enemy.<stat>`, and Runner + Tank become **pure data rows** on the existing melee FSM — no new behavior code. Flyer is the only genuinely new branch. Placeholder-first throughout: reskin existing zombie art with a tint + scale to read as variants now, real per-type art dropped in later via the swap-point.

## Context
The FSM (PATROL → CHASE → ATTACK → HURT → DEAD, plus RETREAT for the perch case) already branches on `this.enemyProfile`, and P3.1 proved a second profile (`ranged`) slots in cleanly. But the branch is shallow: `Enemy.preUpdate` destructures the shared `CONFIG.enemy` tuning (`moveSpeed`, `chaseSpeed`, `detectionRadius`, `attackRange`, `attackCooldown`, …) and only *ranged* substitutes `this.def.*` overrides (`isRanged ? this.def.moveSpeed : moveSpeed`). So a melee enemy can't currently be "faster" or "tankier" from data — every melee zombie is mechanically identical.

The rest of the foundation is ready:
- **`ENEMIES[type]`** is a per-type table; `spawn(x, y, type)` reads `CONFIG.ENEMIES[type] ?? { enemyProfile:'melee' }` into `this.def`. Per-type `body` + `artScale` already work (the Spitter uses them).
- **Enemies are placed** from `CONFIG.LEVEL.enemies` (`[{ type, x, y }]`) via `this.enemies.get(...)` + `enemy.spawn(...)`; the debug right-click spawns a random `Zombie_1..4`.
- **Placeholder pattern** is established (P3.1): a type with no real sheet falls back to a tinted blob, art swaps in later via `TEXTURE_MAP`.
- **Salvage** (P3.3) already flows off any kill via `awardSalvage(this.type, …)`, reading `ENEMIES[type].salvageDrop ?? CONFIG.enemy.salvageDrop`. New types just set (or inherit) a `salvageDrop`.

The design-smell test from `CLAUDE.md` applies directly: *if adding a new enemy requires touching the FSM, that's a smell.* Runner and Tank must NOT — they're stat rows. Flyer legitimately needs code because "ignores gravity and platforms" is new movement, not a new stat. That split is the milestone.

## Scope guard
**In P3.4:**
- **Generalize stat resolution** so every profile resolves per-type overrides: `this.def.<stat> ?? CONFIG.enemy.<stat>` for `maxHealth`, `moveSpeed`, `chaseSpeed`, `detectionRadius`, `attackRange`, `attackCooldown`, `touchDamage` (and existing `patrolRange` etc.). The four `Zombie_N` rows stay minimal and inherit the shared defaults unchanged (regression-safe).
- **Runner** — `enemyProfile:'melee'`, **data only**: low `maxHealth`, high `chaseSpeed`, small `artScale`, quick `attackCooldown`. Reads as a fast, fragile rusher.
- **Tank** — `enemyProfile:'melee'`, **data only**: high `maxHealth`, low `moveSpeed`/`chaseSpeed`, high `touchDamage`, large `artScale` + a fitted `body`, slow `attackCooldown`. Reads as a slow bullet-sponge that hits hard.
- **Flyer** — a **new `enemyProfile:'flyer'`** branch: `body.setAllowGravity(false)`, homes toward the player on **both axes**, no ground/climb/RETREAT logic (it can reach a perched player — a second answer to the perch alongside the Spitter), deals `touchDamage` on contact, dies via the existing DEAD path.
- **Placeholder-first visuals:** a type may set `sheet: '<existing Zombie_N>'` to reuse that art, plus a `tint` and `artScale` so the variants read distinctly *now* (Runner small + one tint, Tank big + another). Flyer uses a placeholder blob + fallback. Real per-type art is a later swap-point drop.
- **`sheet` override in `spawn()`** — resolve texture + animation keys from `this.def.sheet ?? type` (so `Runner` can render `Zombie_2`'s frames), while `this.type` stays the roster id (for `salvageDrop`/spawn lookups).
- **A mixed encounter** in `CONFIG.LEVEL.enemies` combining the new types with zombies, and the new ids added to the debug right-click list. *(Flyer later removed from L1's placement — see the note under "Changes by file"; it stays defined + in the debug list.)*
- **`salvageDrop`** per new type (Runner cheap, Tank richer), or inherit the melee default.

**NOT in P3.4:**
- No new FSM *states* — Runner/Tank reuse the melee states verbatim; Flyer adds a movement branch, not new states.
- No boss (P3.7), no second biome / zone-triggered spawns (P3.6) — new types drop into the existing Level 1 for now.
- No real per-type character art (placeholder reskin now; AI art later), no per-type SFX (P3.9), no ranged variants beyond the existing Spitter.
- No pathfinding — Flyer homes in a straight line (no obstacle avoidance); acceptable for the open forest.
- No loot/behavior beyond the stat set above (no shielded/splitting/summoner types — later chapters).

## Design decisions
- **Runner + Tank are data, full stop.** After the stat-resolution generalization, they are rows in `ENEMIES` with no code path of their own. The proof (DoD): a *fourth* melee variant would be one more row. If the build finds itself adding a `case`/branch for Runner or Tank, something went wrong — push back.
- **One generalization, not a rewrite.** Change the `preUpdate` locals from "ranged uses `def`, melee uses shared" to "**everyone** uses `this.def.<stat> ?? CONFIG.enemy.<stat>`," resolved once near the top. `spawn()` sets `this.health` from `this.def.maxHealth ?? CONFIG.enemy.maxHealth` (and body/scale as the Spitter already does). Keep the ranged-specific reads (`preferredRange`, `firingRange`) as-is.
- **Flyer is a movement branch, kept minimal.** In `spawn()`, when `enemyProfile==='flyer'`: `this.body.setAllowGravity(false)`. In `preUpdate`, a `flyer` branch: if the player is within `detectionRadius`, set velocity toward the player on both axes at `this.def.flySpeed` (normalize the vector), else idle/hover near spawn; skip all `playerUnreachableAbove`/RETREAT/`climbHeight` logic (it doesn't apply). Contact damage reuses the existing player-touch overlap. DEAD: stop, linger, pool out (no gravity re-enable needed). Re-enable gravity only if a flyer type is ever made to fall on death — out of scope.
- **Placeholder = reskin + tint + scale.** Runner/Tank reuse a zombie `sheet` with a `tint` and distinct `artScale`, so a mixed fight is legible immediately with **zero new art**. Flyer gets a small placeholder blob (a new tiny texture in `BootScene`, or reuse an existing one) + the P3.1-style fallback. All three get real art later through `TEXTURE_MAP`/`sheet` with no logic change.
- **Readability is the bar, not just existence.** The DoD isn't "three types exist" — it's "a mixed group **reads clearly**": Tank visibly bigger + slow, Runner visibly smaller + fast, Flyer off the ground. Tint + scale + speed carry that until real art lands. This is a human-playtest gate, not just an assertion.
- **Placeholder-first (golden rule 4)** and **data-driven expansion (golden rule 6)** are the whole milestone — it's the cleanest test of both since P3.1.

## Verification (harness A–G, then DoD)
Drive via `window.game`; spawn types deterministically through the debug hook or by calling `enemy.spawn(x,y,type)`.
- **A. Build/console:** clean; 0 errors/404s.
- **B. Melee regression:** `Zombie_1..4` are unchanged — same health/speed/damage as before the generalization (they inherit `CONFIG.enemy`). Existing L1 plays identically.
- **C. Runner:** dies in fewer hits than a zombie and closes distance faster (assert resolved `maxHealth`/`chaseSpeed` come from its row, not the shared defaults).
- **D. Tank:** takes far more hits, moves slowly, and deals more contact damage; renders visibly larger (artScale) with a body that still fits.
- **E. Flyer:** `body.allowGravity === false`; homes toward the player on both axes and can damage a player perched on a platform (reaches where a melee zombie gives up); dies via the DEAD path.
- **F. Data-driven proof:** temporarily add a throwaway 4th melee variant as a **pure `ENEMIES` row** (stat + `sheet` + `tint`); it spawns and behaves with **no engine edit**; then remove it.
- **G. Mixed fight + FPS:** the mixed cluster reads clearly (screenshot) and holds `actualFps ≥ 55` with the roster active; salvage drops per type on kill.
- **H. Deliverable:** screenshots (each new type; the mixed cluster showing size/height contrast) + a pass/fail table. Human playtest: the three types must *feel* and *read* distinct in a live fight. On green: add a **P3.4 row to `PROGRESS.md`** (what it delivered + parked follow-ups: placeholder reskins pending real art, Flyer has no pathfinding), then remind me to `git commit -m "P3.4: enemy roster — runner/tank (data) + flyer (new enemyProfile), generalized per-type stats"` — no commit without approval.

## Note
**No PixelLab / art generation in P3.4.** All three types ship on placeholders — behavior and tuning are proven first; real art is a separate later pass (the P3.1 Spitter flow: generate CraftPix-matched frames, assemble baseline-aligned strips, commit under `public/assets/…`, point the row's `sheet`/`TEXTURE_MAP` at it — no gameplay code change). Generating sprites before the size/speed/body are tuned would just get redone; and the P3.4 DoD does not depend on any of it.

Ships with **reskinned placeholders** (tinted, rescaled zombie art + a Flyer blob) — a legible mixed fight with zero new art. The parked follow-ups are clean later drops: real per-type sprites via the swap-point, per-type SFX (P3.9), and Flyer pathfinding/obstacle-avoidance if the second biome's geometry ever needs it. After this lands, the enemy system is proven **fully data-driven for stat variants** and extensible by one new `enemyProfile` branch for a genuinely new movement — which is exactly what P3.7's boss will lean on.
