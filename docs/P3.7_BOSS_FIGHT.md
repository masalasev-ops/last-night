# P3.7 — Boss fight + chapter payoff (Boss 1)

> The Chapter 1 climax. A **two-phase boss** — telegraphed intro, health-threshold phases, two attack patterns (melee lunge + ranged/AoE), a boss health bar, a death sequence — fought in a **boss-arena level** reached after Level 2, reusing the whole `GameScene`/player/weapons/HUD/save stack. On the boss's death, a **data-driven `beatOnDefeat`** fires the chapter's story beat (for the demo: a **clue + a surviving ally** who points onward) → **Victory**, which supersedes the P3.6 `EndScene` placeholder. Behavior-first: the boss ships on a **placeholder silhouette**, real AI-generated art dropped in later via the swap-point. The bar is data (phase thresholds + attack params); the boss *entity* is code (a phase controller). Because a later chapter's boss can set `beatOnDefeat: 'rescue'`, the payoff is a reusable, data-selected mechanism — not a one-off.

## Context
P3.6 left the exact seam: Level 2's `nextLevelId: null` routes the shop's Continue to `EndScene` ("Chapter 1 — to be continued"), and the code comments call that "the seam P3.7 replaces." Everything the boss needs already exists:
- **The FSM** (`PATROL/CHASE/ATTACK/HURT/RETREAT/DEAD`) and the melee-attack + `takeDamage` plumbing — the shared movement/damage patterns the boss borrows.
- **`AcidProjectile` + `GameScene.spawnAcid(enemy, player)`** — a working arcing ranged attack that reaches a player at range/height. The boss's ranged/AoE pattern reuses/extends this rather than inventing a projectile.
- **The kill hook** — `Enemy.takeDamage()`'s lethal branch already calls `this.scene.awardSalvage(...)`; the boss's lethal path calls a parallel `this.scene.onBossDefeated(boss)` instead.
- **`UIScene`** already owns the player health bar (`healthFill`) — the boss bar is a conditionally-shown sibling.
- **The level loader** (`CONFIG.LEVELS[levelIndex]`, `this.level`) — the boss arena is a `LEVELS` entry, so the boss fight inherits terrain, camera, HUD, weapons, upgrades, and the death→restart save model **for free**.
- **Run state** carries the player's unlocks/upgrades into the arena (they shop after L2, then fight the boss kitted up).

So the boss is **not** a separate engine — it's a special entity in a boss-arena level, plus a phase controller, a bar, and a payoff beat. The genuinely new code is contained.

## Scope guard
**In P3.7:**
- A **boss-arena level** (`LEVELS` entry, e.g. `3`) — a contained arena (bounded, no `endMarkerX` traversal win), reached via `LEVELS[2].nextLevelId = 3`. Its completion is **boss defeated**, not reach-the-marker.
- A **`Boss` entity** (dedicated class; borrows shared facing/`takeDamage`/attack helpers, **not** in the pooled `enemies` group) with:
  - a **telegraphed intro** (boss appears/roars/steps in; a brief scripted lock before the fight opens),
  - a **data-driven phase controller** — `phases: [{ hpAbove, … }]` selected by current HP fraction (two phases for the demo: P1 → transition at a threshold → P2, each with its own attack set / cadence / speed),
  - **two attack patterns** — a **melee lunge** (telegraph → dash/slam → recover, extending the melee attack) and a **ranged/AoE** (reusing `AcidProjectile`/`spawnAcid`, e.g. a multi-glob volley or a telegraphed ground AoE), each **telegraphed** so the player can react,
  - a **death sequence** (a distinct, non-pooled death — collapse/flash, brief pause) that ends by firing the beat.
- A **boss health bar** — a wide bar (top or bottom), shown only while a boss is active (`UIScene`), reading the boss's HP fraction; phase transitions optionally flash/segment it.
- A **`beatOnDefeat` story-beat system** — the boss level declares `beatOnDefeat` in data; `onBossDefeated` fires it. For the demo: **`clueAndAlly`** → a short scripted beat (a surviving ally figure + 2–3 lines of dialogue pointing onward + a clue line) → a **Victory** screen (chapter-complete + hook). Data-driven so a later boss can select a different beat.
- **Wiring**: L2 → shop → Continue → **boss arena** → boss dies → beat → Victory. `EndScene` is superseded by the Victory beat (kept or repurposed).
- **No soft-locks** — intro, phase transition, and death are handled so the player is never stuck (input restored after intro; a boss that dies mid-attack cleanly resolves; the beat always reaches Victory).
- **Placeholder-first** — the boss is a large **placeholder silhouette/blob** (behavior-first, à la the P3.1 Spitter blob / Flyer blob); the ally is a placeholder figure; the clue is inline text. Real AI art (PixelLab, CraftPix-matched) and any NPC art drop in later via the swap-point.

**NOT in P3.7:**
- **No real boss or ally art** — placeholders only; the swap-point drop is a later art pass (the boss is a grounded creature, so PixelLab suits it — but that's not this milestone).
- **No audio** (P3.9) — the intro/attacks/death are silent; SFX/music is a later pass.
- **No real menu/options** (P3.8) — Victory routes to the existing Title.
- **No rescue mechanic** — the demo beat is **clue + survivor hook**, deliberately *not* a rescue (rescues are reserved for later chapters where they're earned). `beatOnDefeat` leaves room for it.
- **No note-collectible / dialogue-tree system** — the clue is a scripted line, not a full note-reader or branching dialogue (those are later polish; the plan's "note-reader" is deferred).
- **No second boss, no boss variants, no difficulty modes.**
- No new player weapons/upgrades/enemies (the boss fights the existing arsenal).

## Design decisions
- **Boss = a dedicated entity in a boss-arena level, not a new scene and not a pooled enemy.** Reusing `GameScene` (terrain, camera, HUD, weapons, save/death-restart) means the fight inherits the whole stack — the player's upgrades apply, dying restarts the arena from its start (P3.5's model), and the bullet/`onBulletHitEnemy` damage path already works on it. A **standalone `Boss` class** (borrowing facing/`takeDamage`/the melee + `spawnAcid` calls) keeps boss-only logic (phases, intro, death, beat trigger) out of the pooled `Enemy` FSM — a boss is a genuinely different thing, not a stat variant, so a dedicated entity is correct (the design-smell test cuts the other way here vs. Runner/Tank).
- **Phases are data; the controller is code.** A `BOSS` config carries `maxHealth` and `phases: [{ hpAbove, attacks: [...], moveSpeed, attackCadence, … }]`. The controller picks the phase whose `hpAbove` the current HP fraction clears, and switches attack set/cadence on transition (with a brief telegraphed transition beat). Two phases for the demo; a third is a data row.
- **Reuse the ranged system.** The boss's ranged/AoE pattern is built on `AcidProjectile`/`spawnAcid` (proven to arc onto the player) — a volley or a telegraphed AoE, not a new projectile class. Melee lunge extends the existing melee attack (telegraph → commit → recover). Keeps new code to the *orchestration* (which attack, when), not new physics.
- **Everything telegraphs; nothing one-shots unfairly.** Each attack has a wind-up the player can read and dodge; phase 2 is faster/denser, not a stat cliff. This is the "no soft-lock / fair fight" half of the DoD — a boss that's unreadable or inescapable fails it as much as one that literally locks.
- **The boss level's win condition is boss-death, not `endMarkerX`.** `GameScene` detects a `LEVEL.boss` and routes completion through `onBossDefeated` → `beatOnDefeat`, bypassing the `endMarkerX → enterShop` path. No shop after the boss; the arena is bounded so the player can't walk past it.
- **`beatOnDefeat` is the reusable payoff seam.** `onBossDefeated` reads the level's `beatOnDefeat` and fires the matching beat. `clueAndAlly` (demo) = a scripted overlay/scene: ally figure + dialogue + clue → Victory. The data indirection is the whole point — Chapter 2's boss sets `rescue` and the same hook fires a different set-piece, no engine change.
- **Death/restart economics stay P3.5.** Dying in the arena restarts the arena from its start with the loadout you entered it with (the checkpoint was taken on Continue into the boss level). No mid-fight checkpoint. The boss resets to full on a retry (arena re-`create()`s).
- **Placeholder-first (golden rule 4).** The boss is a big tinted placeholder with a simple attack-telegraph tell (a flash/scale before each attack) so the fight is *readable* without art; the ally is a placeholder figure + text. Real art is a clean later swap on the boss's texture key.
