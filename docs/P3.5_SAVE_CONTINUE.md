# P3.5 — Save / Continue (level-boundary checkpoints)

> Fifth milestone of Phase 3. Turn the in-memory `RunState` into a **persisted** one: a run's progress — salvage, unlocked weapons, owned upgrades, and which level you're on — survives a browser reload via **Continue**, and **New Game** wipes it. The deliberate design (your call): **the only checkpoint is a level boundary.** Clearing a level banks your salvage and saves; **dying restarts the current level from its start with the loadout you entered it with** — salvage picked up in the failed attempt is *not* banked. There is no mid-level save. This is the enabling payoff of P3.3's design: because upgrades live in `ownedUpgrades` and apply via `recompute()`, a load is just "restore the fields + `recompute()`" — no bespoke deserialization of the weapons table.

## Context
P3.3 built `RunState` as an in-memory singleton that already survives scene transitions, and its docstring literally notes it's "P3.5-load-ready (restore = set fields + `recompute()`)." What's missing is the persistence layer and a level cursor:
- `runState` today holds `salvage`, `unlockedWeapons`, `ownedUpgrades`, and the derived `weapons` table; it has `reset()` (seeded from `CONFIG`) and `recompute()` (rebuild `weapons` from the template + owned upgrades). **`reset()` has existed unused since P3.3** — this is where it finally gets wired.
- There's **no `levelIndex`** and **no localStorage** anywhere yet.
- **Boot flow**: `BootScene.create()` → `runState.init()` → `this.scene.start('Game')` (auto-starts gameplay; no title).
- **Death**: `GameScene.update()` → on `player.wantsRestart` → `stop('UI')` + `scene.restart()` (keeps the current in-memory `runState`).
- **Win**: `stop('UI')` + `start('Shop')`. **Shop Continue**: `this.scene.start('Game')` (stub-restarts Level 1 until P3.6).

So P3.5 adds: a versioned save in localStorage, a `levelIndex` + `phase` cursor, a **checkpoint** captured at each level boundary (revert target on death), and a **minimal Continue / New Game** entry point. Note: localStorage is the correct tool here — this is the real Vite build in the player's own browser, **not** a sandboxed artifact (the "no localStorage in artifacts" rule doesn't apply to the game).

## Scope guard
**In P3.5:**
- **Persisted save** (versioned) in localStorage: `{ version, levelIndex, phase, salvage, unlockedWeapons, ownedUpgrades }`. The derived `weapons` table is **not** stored — it's rebuilt by `recompute()` on load.
- **`RunState` persistence API**: `save()`, `loadFromStorage()`, `hasSave()`, `clearSave()`, `newGame()`, `checkpoint()`, `restoreCheckpoint()`, plus a private `serialize()`/`applySnapshot()` pair. All localStorage access wrapped in `try/catch`.
- **Level-boundary checkpoint model** (the challenge design):
  - **New Game / Continue-into-a-level / advance-to-next-level** → take a checkpoint (in-memory snapshot **and** `save()`), then start the level. This snapshot is "the start of this level."
  - **Death** → `restoreCheckpoint()` (revert `runState` to the level-start snapshot, discarding the attempt's salvage) → restart the level. **No save on death.**
  - **Level clear** → bank salvage, `save()` with `phase:'shop'`, open the shop.
  - **Each shop purchase** → `save()` (unlocks/upgrades persist even if the browser closes mid-shop).
  - **Continue (shop)** → `levelIndex++`, `phase:'level'`, checkpoint + `save()`, start the next level.
- **A minimal Title/menu scene**: **Continue** (only if a valid save exists) resumes at the saved `levelIndex`/`phase`; **New Game** wipes the save, `reset()`s, and starts Level 1. Text/placeholder UI — the real title screen is P3.8.
- **Graceful save handling**: a missing / unparseable / wrong-`version` save is treated as **no save** (Continue hidden, fall back to New Game) — never a crash. A failed `setItem` (private mode / quota) is swallowed; in-session play (including death-revert) still works via the in-memory checkpoint.
- **HUD salvage stays the in-memory value** — you watch it climb during a level, and it visibly reverts on death (that tension is intended; it's what "banked only on clear" *feels* like).

**NOT in P3.5:**
- **No mid-level / mid-run save** — by design. The finest granularity is one level. (Deliberately omitted so you can playtest the difficulty; trivial to add later if level-only feels too punishing.)
- **No real title/menu art or options** (volume, controls, etc.) — P3.8. This is a two-button placeholder.
- **No Level 2** (P3.6). `levelIndex` is persisted and advances on Continue, but the level *loaded* is always the Level-1 stub until P3.6 wires `levelIndex` → level data. The `phase` cursor still makes "closed while in the shop" resume correctly.
- **No per-run stats / meta-progression / achievements / multiple save slots** — single slot, single run.
- **No checkpoint mid-level, no "retry from shop" on death** (death sends you to the level start, not back to the shop — that's the point).
- No settings persistence, no cloud save, no export/import.

## Design decisions
- **The checkpoint is a level boundary, full stop.** `runState` is authoritative *during* a level (salvage climbs in memory), but the **snapshot** used for death-revert and the **save** written to disk are only taken when a level begins, when a level is cleared, on shop purchases, and on Continue. Nothing writes mid-level. Death reverts to the most recent level-start snapshot. This is the whole challenge model — dying costs you the current level's salvage and progress, and you can't save-scum.
- **In-memory checkpoint + on-disk save are separate concerns, kept in sync.** `save()` serializes and (a) stores the snapshot on `runState` as the in-memory checkpoint **and** (b) writes localStorage. **Death-revert reads the in-memory checkpoint** (always available, even if localStorage is blocked); **cross-session Continue reads localStorage**. This makes death-revert robust regardless of storage availability, and degrades gracefully (lose only cross-session persistence, never in-session play).
- **Load = set fields + `recompute()`** — exactly what P3.3 was built for. `applySnapshot()` sets `salvage`/`unlockedWeapons`/`ownedUpgrades` (rehydrate the `Set` from the stored array)/`levelIndex`/`phase`, then `recompute()` rebuilds the `weapons` table (so upgraded stats are live) and the in-memory checkpoint is set. No weapon-table serialization.
- **Versioned + defensive.** `SCHEMA_VERSION` constant; `SAVE_KEY = 'lastnight.save.v1'`. On load: `try` parse → check `version === SCHEMA_VERSION` and shape → else return `false` (no save). Any throw (blocked storage, bad JSON) → `false`. A `false` load means the Title shows New Game only. Never crash on a stale or corrupt save.
- **Phase cursor resolves the "closed in the shop" case.** `phase:'level'` → Continue starts `Game` at `levelIndex`; `phase:'shop'` → Continue opens the `Shop` (salvage already banked at clear, so no progress lost if they closed before buying). Two values, one flag.
- **Minimal Title scene, real menu deferred — this is the confirmed entry point (not auto-Continue).** Boot no longer auto-starts `Game`; it routes to a text `TitleScene` with two choices. **Continue** appears only when `hasSave()` is true, and its label names the checkpoint it will resume — e.g. `Continue — Level 2` (`phase:'level'`) or `Continue — Shop (Level 1 cleared)` (`phase:'shop'`) — read from the saved `levelIndex`/`phase` without loading the full run. **New Game** is always present. Placeholder-first — P3.8 replaces it with the real thing, same routing.
- **`levelIndex` is plumbed now, consumed by P3.6.** It advances and persists this milestone; the actual `levelIndex → level data` lookup is P3.6's job. Building the cursor now means P3.6 is a data wiring, not a save-format change.

## Changes by file
1. **`src/runState.js`**
   - Add fields: `levelIndex` (default `1`), `phase` (`'level'`), and a private `_checkpoint` (in-memory snapshot).
   - Add consts `SAVE_KEY = 'lastnight.save.v1'`, `SCHEMA_VERSION = 1`.
   - `serialize()` → `{ version: SCHEMA_VERSION, levelIndex, phase, salvage, unlockedWeapons: [...], ownedUpgrades: [...this.ownedUpgrades] }`.
   - `applySnapshot(snap)` → set `salvage`/`levelIndex`/`phase`/`unlockedWeapons`; `ownedUpgrades = new Set(snap.ownedUpgrades)`; `recompute()`; set `_checkpoint = serialize()`.
   - `save()` → `_checkpoint = serialize()`; `try { localStorage.setItem(SAVE_KEY, JSON.stringify(_checkpoint)) } catch {}`.
   - `loadFromStorage()` → `try` read+parse; validate `version` + shape; `applySnapshot(parsed)`; return `true`; on any failure return `false`.
   - `hasSave()` → `true` only if the stored value parses and `version === SCHEMA_VERSION`.
   - `clearSave()` → `try { localStorage.removeItem(SAVE_KEY) } catch {}`.
   - `checkpoint()` → `save()` (snapshot + persist; called at level-start boundaries).
   - `restoreCheckpoint()` → `applySnapshot(_checkpoint)` (revert to level start; used on death). No disk write.
   - `newGame()` → `reset()`; `levelIndex = 1`; `phase = 'level'`; `clearSave()`; `checkpoint()`.
   - `advanceLevel()` → `levelIndex++`; `phase = 'level'`; `checkpoint()`.
   - `enterShop()` → `phase = 'shop'`; `save()` (bank the cleared level's salvage).
   - Keep `init()` but it no longer auto-starts a run — the Title decides New Game vs Continue.
2. **`src/scenes/TitleScene.js` (NEW)**
   - Minimal text menu: game title + **Continue** (rendered only if `runState.hasSave()`) + **New Game** (always).
   - The **Continue** label names the checkpoint, read from a lightweight peek at the save (`levelIndex` + `phase`) without a full load — `Continue — Level {N}` for `phase:'level'`, `Continue — Shop (Level {N} cleared)` for `phase:'shop'` — so the player knows what they're resuming. Add a `runState.peekSave()` helper returning `{ levelIndex, phase } | null` (parses + version-checks, no state mutation) for the label.
   - Continue → `runState.loadFromStorage()`; if it returns `false`, fall through to New Game; else route by `phase`: `'shop'` → `start('Shop')`, `'level'` → `start('Game')`.
   - New Game → `runState.newGame()` → `start('Game')`. (If a save exists, no confirm prompt in P3.5 — a "New Game overwrites your run" confirm is a P3.8 menu nicety.)
   - Placeholder styling (matches the Shop's text UI).
3. **`src/scenes/BootScene.js`**
   - Keep asset generation + `runState.init()`; change the tail from `start('Game')` to **`start('Title')`**.
4. **`src/scenes/GameScene.js`**
   - **Death** (`player.wantsRestart`): `runState.restoreCheckpoint()` **before** `stop('UI')` + `scene.restart()` — the restarted level rebuilds `Player` from the reverted `runState` (salvage back to level-start, upgrades/unlocks intact).
   - **Win**: before `start('Shop')`, call `runState.enterShop()` (bank salvage + save `phase:'shop'`). Keep `stop('UI')`.
   - (No checkpoint call in `create()` — the checkpoint is taken by the transition *into* the level: `newGame`/`advanceLevel`/`loadFromStorage`.)
5. **`src/scenes/ShopScene.js`**
   - After each successful `unlockWeapon`/`buyUpgrade`, call `runState.save()` (persist purchases immediately).
   - **Continue**: `runState.advanceLevel()` then `this.scene.start('Game')` (levelIndex persisted; stub still loads Level 1 until P3.6).
6. **`src/main.js`**
   - Register `TitleScene` in the scene list. Expose it in the dev block if useful. (`window.runState` already exposed for the harness.)

## Verification (puppeteer-core harness A–H, then DoD)
Drive via `window.runState` + `window.game`; simulate a "reload" by re-running `runState.init()` then `loadFromStorage()` (a fresh module state reading the same localStorage), and inspect fields + `weapons` after `recompute()`.
- **A. Build/console** — clean; 0 errors/404s.
- **B. New Game writes a save** — from no save, New Game seeds defaults (salvage 0, rifle-only, no upgrades, `levelIndex 1`, `phase 'level'`) and `localStorage[SAVE_KEY]` exists with `version === 1` and that shape.
- **C. Persist + Continue across a reload** — clear L1 and buy `rifle_dmg_1`; reload (re-init + `loadFromStorage`); `salvage`/`unlockedWeapons`/`ownedUpgrades`/`levelIndex`/`phase` all restored, and `recompute()` gives `weapons.rifle.damage` the upgraded value (the P3.2/P3.3 seam survives serialization).
- **D. Phase routing** — a save with `phase:'shop'` → Continue opens `Shop`; `phase:'level'` → Continue starts `Game`.
- **E. Death reverts to the checkpoint (the challenge core)** — enter L1 with some banked salvage, earn more, then die: `salvage` reverts to the level-start value (attempt's earnings gone), `unlockedWeapons`/`ownedUpgrades` unchanged, and the level restarts; **no new save was written on death** (the on-disk save still shows the pre-death checkpoint).
- **F. New Game wipes** — with a save present, New Game → `hasSave()` false afterward, `runState` back to defaults, L1 starts.
- **G. Corrupt / version-mismatch / blocked storage** — a garbage or `version:0` value → `hasSave()` false and `loadFromStorage()` returns `false` (Title shows New Game only), no crash; and with `setItem` forced to throw, `save()` is swallowed and in-session death-revert still works via `_checkpoint`.
- **H. FPS/round-trip** — Title → Game → Shop → Game and death-restart leave no leaked scenes/listeners; `actualFps ≥ 55` in-level.
- **I. Deliverable** — screenshots (Title with Continue + New Game; a resumed run showing banked salvage + an applied upgrade; salvage reverting on a death) + pass/fail table. Human playtest: clear a level, buy something, reload → Continue lands you mid-progress; die in a level → you redo it from the start and the run's meta-progress (unlocks/upgrades) is intact but the attempt's salvage is gone. ## Note
Ships with a **two-button text Title** and a **single versioned save slot** — enough to prove Continue / New Game and the level-boundary challenge model, with real menus (P3.8) and `levelIndex→level` wiring (P3.6) as clean later drops on the same cursor. The **level-only save is a deliberate difficulty choice** — playtest it; if dying-redoes-the-level feels too harsh, a mid-level checkpoint is a small additive change (snapshot at a mid-level trigger, same `restoreCheckpoint()` path) that doesn't disturb this format. After this lands, a run is durable: the loop is **kill → salvage → shop → stronger → survive → saved**, and P3.6 can add a second level knowing progress already persists across it.
