# P3.3 — Salvage, end-of-level shop & upgrades

> Third milestone of Phase 3. Enemies drop **salvage**; on level-complete an **end-of-level shop scene** spends it on **weapon unlocks** and **data-driven upgrades** (damage / reload / mag / fire-rate stat modifiers); the effects **persist into the next level**. The enabling piece is a **`RunState` module** that owns run-scoped state and a **runtime weapons table** — a mutable clone of `CONFIG.WEAPONS` that upgrades modify, which `Player.get weapon()` now reads instead of `CONFIG`. Because P3.2 already resolves damage/range from the live weapon at fire time and stamps it on the bullet, an upgrade to a stat takes effect on the very next shot with **zero engine change** — that seam is the whole reason P3.2 was built the way it was. Persistence here is **in-memory across scenes** (a module singleton); `localStorage` save/continue is P3.5.

## Context
P3.2 left the weapon system exactly where P3.3 needs it: `Player.get weapon()` returns `CONFIG.WEAPONS[this.currentWeaponId]` **live** (never snapshotted), and `Bullet.fire()` **stamps** damage/range/tint at fire time. So the only thing standing between "static weapons" and "upgradeable weapons" is *what the live lookup reads from*. Redirect it from the immutable `CONFIG.WEAPONS` template to a mutable per-run table, and upgrades flow through untouched.

The rest of the foundation:
- **`CONFIG.ENEMIES`** is already a per-type table. `Spitter` carries a currency-drop field committed in P3.1 as `scrapDrop: { min: 2, max: 4 }` (commented "unused until P3.3") — P3.3 **renames it to `salvageDrop`**; the four melee `Zombie_N` rows have only `aiProfile` and need a `salvageDrop` (or a shared default).
- **The kill hook** is a single place: `Enemy.takeDamage()`'s lethal branch (`this.health <= 0 → STATE.DEAD`). The enemy already calls scene methods (`this.scene.spawnAcid(...)`), so awarding salvage from there is idiomatic.
- **Level-complete** today: reaching `CONFIG.LEVEL.endMarkerX` sets `player.won = true`; `UIScene` shows "YOU MADE IT / Press R to replay"; `R` → `scene.restart()`. P3.3 replaces the R-to-replay dead-end with a transition into the shop, then on into the next level.
- **Scenes**: `main.js` registers `[BootScene, GameScene, UIScene]`; `UIScene` is launched alongside `GameScene`. P3.3 adds a `ShopScene`.
- There is **no currency, no upgrade, no unlock gating** yet — P3.2 shipped all three weapons switchable. P3.3 introduces the economy, which means gating two of them behind purchases (see decisions).

The real core of P3.3 is therefore **not the shop UI** — it's the **`RunState` seam** (a run-scoped source of truth for weapons/salvage/unlocks that outlives any single scene) plus the **win → shop → next-level loop**. Get those right and the shop is a data-driven list on top.

## Scope guard
**In P3.3:**
- A **`RunState` module** (`src/runState.js`) — the single owner of run-scoped state: `salvage`, `unlockedWeapons`, `ownedUpgrades`, and a **runtime `weapons` table** (a deep clone of `CONFIG.WEAPONS`). Survives scene transitions because it's a module singleton, not scene state. Exposes a `reset()` (for New Game, wired in P3.5) and a `recompute()` (rebuild `weapons` from `CONFIG.WEAPONS`, then apply all owned upgrades).
- **`Player.get weapon()` reads `runState.weapons[id]`** instead of `CONFIG.WEAPONS[id]`. `CONFIG.WEAPONS` becomes the immutable **template**; `runState.weapons` is the live, upgrade-modified copy.
- **Salvage**: enemies award `salvageDrop.min..max` on death (auto-collected, with a floating `+N` text for feedback); a HUD salvage counter.
- **`UPGRADES` table** (data): one-shot stat modifiers (`add`/`mult`) targeting a weapon stat (`damage`/`reloadTime`/`magSize`/`fireRate`), with an optional `prereq` for tiers. **Prove one full tiered path end-to-end** (e.g. Rifle Damage I → II) plus a couple of single-tier upgrades on other weapons/stats.
- **Weapon unlocks**: the player starts with **Rifle only** unlocked; **Shotgun + SMG are shop purchases**. `runState.unlockedWeapons` gates switching (keys 2/3 and `switchWeapon` no-op for a locked id).
- **`ShopScene`**: a between-levels scene reading `RunState` — salvage balance, a purchasable list (weapon unlocks + upgrades) with costs, click-to-buy (disabled when unaffordable / already owned / prereq unmet / already unlocked), and a **Continue** button that starts the next level.
- **The loop**: level-complete → `ShopScene` → next level, with `RunState` (salvage, unlocks, applied upgrades) **persisting across the transition**.
- **`recompute()` application model** — upgrades are applied by rebuilding from the template and re-applying all owned upgrades, never by mutating in place. Deterministic, stack-safe, and P3.5-load-ready.

**NOT in P3.3:**
- **`localStorage` save/continue, checkpoints, New Game wipe** (P3.5). `RunState.reset()` exists but nothing calls it yet — persistence this milestone is purely in-memory across scenes.
- **`target: 'player'` upgrades** (maxHealth/armor, etc.) — a reserved schema value with **no consumer** this milestone (like `fireMode:'burst'`); the DoD's stat list is weapon-only. Player stats stay sourced from `CONFIG`. Adding a `runState.player` seam is a small later step.
- **Physical collectible salvage entities** (walk-over pickups like the health chest) — salvage is auto-collected on kill; a floating `+N` is the only feedback. Collectible drops are optional later polish.
- **Real shop art / layout** — functional text UI now (placeholder-first); a styled shop is a later art/UX pass.
- **Level 2** (P3.6). Until it exists, the shop's **Continue restarts Level 1 as a stub**; the DoD's "persists into the next level" is proven by `RunState` surviving the `Game → Shop → Game` transition (salvage kept, weapons still upgraded, unlocks still unlocked), not by a distinct second level.
- Mid-level spending; multiple shops per level; selling/refunds; repeatable (infinite-stack) upgrades; per-weapon character art; audio (P3.9); menus (P3.8).

## Design decisions
- **`RunState` is the run-scoped source of truth, `CONFIG` is the template.** `CONFIG.WEAPONS`/`CONFIG.UPGRADES` are never mutated. At run start `runState.recompute()` deep-clones `CONFIG.WEAPONS` into `runState.weapons`, then applies every id in `runState.ownedUpgrades`. `Player.get weapon()` reads `runState.weapons[id]`. This keeps the template pure (New Game / P3.5 load just rebuild from it), makes upgrade stacking deterministic, and is the exact realization of P3.2's "read live, never snapshot" — we only move *where* "live" points.
- **Upgrades apply via full recompute, not in-place mutation.** `buyUpgrade(id)` adds the id to `ownedUpgrades`, deducts cost, then calls `recompute()`. `recompute()` = clone template → for each owned upgrade, apply `mode:'add'` then `mode:'mult'` (adds before mults, so order of purchase never changes the result). One-shot (an owned upgrade can't be re-bought). This avoids the classic "mult applied twice / add-after-mult" bugs and means P3.5 can restore a save by setting `ownedUpgrades` and calling `recompute()`.
- **Tiers via `prereq`.** An upgrade with `prereq: 'rifle_dmg_1'` is unbuyable (and hidden or greyed) until that id is owned. Prove one two-tier path end-to-end so the gate is exercised.
- **Weapon unlocks gate switching.** `runState.unlockedWeapons` starts `['rifle']`. `switchWeapon(id)` and the 1/2/3 loop skip ids not in it; the HUD only cycles unlocked weapons. `player.ammo` still holds a mag entry per weapon id (built from `CONFIG.WEAPONS`), but a locked weapon simply can't be selected. A dev flag (`runState.unlockAll()` behind the existing debug toggle) unlocks all three for testing. *(This is the one behaviour change from P3.2, which shipped all three switchable — now two are earned.)*
- **Salvage is awarded on the kill, auto-collected.** `Enemy.takeDamage()`'s lethal branch calls `this.scene.awardSalvage(this.type, this.x, this.y)`, which rolls `min..max` from `CONFIG.ENEMIES[type].salvageDrop` (falling back to a shared default for any type without one), adds to `runState.salvage`, spawns a floating `+N` text, and refreshes the HUD. Single kill hook, no pickup physics.
- **Mags refill full at level start.** Leaving the shop starts the next level with every unlocked weapon's mag full — avoids weird partial-mag states after a `magSize` upgrade, and matches the genre (you re-kit between levels). Reload behaves normally *within* a level. Since a fresh `Player` is constructed per level and builds mags from the (now runtime) `magSize`, this falls out naturally.
- **Scene flow.** On `player.won`: stop `GameScene` + `UIScene`, `start('Shop')`. `ShopScene.Continue` → `start('Game')` for `nextLevelId` (stub = restart L1 until P3.6), which relaunches `UIScene` as it does today. `RunState` is untouched by any of these, so it carries forward. Death/restart is left as-is for now (keeps `RunState`); real death/checkpoint economics are P3.5.
- **Placeholder-first (golden rule 4).** Text-label shop, text salvage counter, floating `+N` — all functional, no art. Real shop layout, weapon icons, and collectible salvage drops slot in later with no logic change.

## Changes by file
1. **`src/config.js`**
   - **Rename the existing committed `Spitter.scrapDrop` → `salvageDrop`** (P3.1 shipped it as `scrapDrop`; also update its "salvage economy" comment). Then add **`salvageDrop`** to the four melee rows (or a shared `CONFIG.enemy.salvageDrop` default consulted when a type omits it) — e.g. melee `{ min: 1, max: 2 }`; keep `Spitter`'s `{ min: 2, max: 4 }`.
   - Add an **`UPGRADES`** table, ids keyed, matching the schema `{ target, stat, mode, amount, cost, prereq? }`. Seed a small set that proves the mechanics: a two-tier Rifle damage path (`rifle_dmg_1` add, `rifle_dmg_2` add with `prereq: 'rifle_dmg_1'`), a Shotgun reload `mult` (<1 = faster), an SMG `magSize` add, a Rifle `fireRate` mult. `target` = a weapon id (`'player'` reserved, unused).
   - Add a **`SHOP`** object: `{ weaponsForSale: ['shotgun','smg'], upgradesForSale: [<upgrade ids>] }`.
   - Add run-start defaults: `startingWeapon: 'rifle'` (also `defaultWeaponId` already exists) and a `salvageStart: 0`. Leave `WEAPONS` as the immutable template (add a comment saying so).
2. **`src/runState.js` (NEW)**
   - A singleton object + functions (module-level state, mirroring the `config.js` import style). Fields: `salvage`, `unlockedWeapons` (array/Set, seeded `['rifle']`), `ownedUpgrades` (Set), `weapons` (runtime table).
   - `init()` / `reset()` — seed defaults from `CONFIG`, then `recompute()`. (`reset()` is the New-Game entry P3.5 will call.)
   - `recompute()` — `weapons = structuredClone(CONFIG.WEAPONS)`; for each owned upgrade apply `add` then `mult` to `weapons[u.target][u.stat]`.
   - Helpers: `addSalvage(n)`, `canAfford(id)`, `isUnlocked(id)`, `unlockWeapon(id)` (checks cost/affordability, deducts, pushes to `unlockedWeapons`), `buyUpgrade(id)` (checks affordability + prereq + not-owned, deducts, adds to `ownedUpgrades`, `recompute()`), `upgradeAvailable(id)` (affordable && prereq met && not owned). Include `unlockAll()` for the debug path.
   - `init()` is called once at boot (BootScene or main) so `runState.weapons` exists before the first `Player`.
3. **`src/entities/Player.js`**
   - `get weapon()` → `runState.weapons[this.currentWeaponId]` (was `CONFIG.WEAPONS[...]`).
   - `currentWeaponId` defaults to the first **unlocked** weapon (`runState.unlockedWeapons[0]`, i.e. rifle).
   - `switchWeapon(id)`: additionally no-op if `!runState.isUnlocked(id)`. The 1/2/3 loop maps to `Object.keys(CONFIG.WEAPONS)` as today but skips locked ids (a locked key press does nothing).
   - `ammo` map still built from `Object.keys(CONFIG.WEAPONS)` using **runtime** `magSize` (so a `magSize` upgrade is reflected when the next level's `Player` is built — mags start full).
   - (No maxHealth change — `target:'player'` is out of scope.)
4. **`src/entities/Enemy.js`**
   - In `takeDamage()`'s lethal branch (right where it sets `STATE.DEAD`), call `this.scene.awardSalvage(this.type, this.x, this.y)`. Nothing else changes.
5. **`src/scenes/GameScene.js`**
   - Add `awardSalvage(type, x, y)`: roll `min..max` from `CONFIG.ENEMIES[type].salvageDrop ?? CONFIG.enemy.salvageDrop`, `runState.addSalvage(n)`, spawn a short-lived floating `+N` text at `(x,y)`, and trigger a HUD refresh.
   - Win block: when `player.won` becomes true, stop this scene + `UI` and `this.scene.start('Shop')` (instead of parking on R-to-replay). Ensure the next level relaunches `UIScene` the same way create() does today.
   - Debug: fold `runState.unlockAll()` into the existing debug toggle so god-mode also unlocks all weapons for testing.
6. **`src/scenes/ShopScene.js` (NEW)**
   - Reads `RunState` + `CONFIG.SHOP`/`CONFIG.UPGRADES`. Renders: salvage balance; a list of weapon unlocks (name, cost, "OWNED"/buy) and upgrades (name, cost, tier/prereq state, "OWNED"/buy); a **Continue** button. Click a buyable row → `runState.unlockWeapon`/`buyUpgrade`, then re-render (balance + row states update). Buttons disabled/greyed when unaffordable, owned, or prereq-unmet. Continue → `this.scene.start('Game')` for `nextLevelId` (stub: Level 1). Functional text UI; no art.
7. **`src/scenes/UIScene.js`**
   - Add a **salvage counter** text (reads `runState.salvage`), refreshed each frame (or on the `awardSalvage` event). Sits with the ammo/weapon readout, styled like the others.
8. **`src/main.js`**
   - Register `ShopScene` in the scene list; call `runState.init()` at boot before the first scene needs `runState.weapons` (or do it in `BootScene.create()`).
